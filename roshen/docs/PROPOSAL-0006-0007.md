# Migrations 0006 & 0007 — Schema / RLS Proposal (review before apply)

Two separate, reviewable migrations. **Nothing applied; no screens built.**
Reuses existing helpers: `is_admin()`, `is_global()`, `my_area_ids()`,
`my_region_ids()`, `app_role()`. Full detail in `WORKSPACE-TASKS.md` and
`REQUESTS-MODULE.md`; this is the concise pre-apply summary.

================================================================
## 0006_workspace_foundation
================================================================

### 1. Tables
`task`, `task_visibility`, `task_comment`, `task_activity`,
`calendar_event`, `notification`, `task_reminder`.

### 2. Key fields
- **task** — `id, company_id, title, description, priority(enum),
  status(enum), start_date?, due_date, due_time?, timezone('Asia/Riyadh'),
  reminder_offset(enum), reminder_at?, completed_at?, assigned_to, created_by,
  visibility(enum), visible_role(app_role)?, related_area_id?, related_branch_id?,
  related_agent_id?, related_import_batch_id?, related_sla_target_id?,
  created_at, updated_at`.
- **task_visibility** — `id, task_id, user_id?, role?, region_id?, area_id?,
  branch_id?, agent_id?` (one grant per row).
- **task_comment** — `id, task_id, author_id, body, created_at`.
- **task_activity** — `id, task_id, actor_id, type, from_value?, to_value?, created_at`.
- **task_reminder** — `id, task_id, remind_at, offset(enum), sent_at?` (multi-reminder; MVP can use task.reminder_at).
- **calendar_event** — `id, company_id, owner_id, title, kind('task'|'trip'|'leave'|'custom'),
  start_date, end_date?, all_day, status_color?, related_task_id?, related_request_id?, created_at`.
- **notification** — `id, company_id, user_id, type(enum), title, message,
  related_task_id?, related_request_id?, action_url?, is_read(false), created_at, read_at?`.
- Enums: `task_priority`, `task_status`, `task_visibility`, `reminder_offset`, `notification_type`.

### 3. Visibility rules (tasks)
A user sees a task iff: creator · assignee · explicit user grant · their role
granted · their scope (area/region/branch/agent) granted · task's related_area in
their scope · `all_managers` + is_global · or is_admin.

### 4. RLS policies
- `task` SELECT `using (can_see_task(task))` (SECURITY DEFINER helper encodes the
  rule above). INSERT `with check (created_by = auth.uid())`; assigning to others
  requires is_global() or assignee-in-scope (server action). UPDATE: creator /
  admin / company_manager full; assignee → status + comments only.
- `task_comment` / `task_activity` / `task_visibility` / `task_reminder`:
  visible iff parent task visible (`task_id in (select id from task)`).
- `calendar_event`: SELECT if owner, or linked task/request visible, or is_global.
- `notification`: SELECT/UPDATE `using (user_id = auth.uid())` — each user only
  their own. Rows are created only for users allowed to see the source task
  (visibility resolved at creation), so notifications respect task visibility.

### 5. Calendar & Notifications connection
- Calendar (MVP) is a **query/view** over `task.due_date` (+ trip/leave dates
  from 0007); `calendar_event` is only for custom events. RLS on the underlying
  rows scopes what appears.
- Notifications: shared `notification` table; top-bar bell unread count =
  `notification where user_id = me and not is_read`. Created by server
  actions/triggers per event (assigned, due-soon, overdue, comment, status…).

### 6. Sidebar / top-bar impact
- Sidebar already shows **Workspace** and **Calendar** (stubs). No new items
  needed for 0006.
- Top-bar **bell** will switch from the placeholder "3" to the live unread count
  once notifications exist.

## 0007_requests_foundation
================================================================

### 1. Tables
`request`, `request_approval`, `request_attachment`, `request_activity`,
`business_trip_request_detail`, `expense_request_detail`, `leave_request_detail`.

### 2. Key fields
- **request** — `id, company_id, request_type(enum), title, requested_by,
  assigned_approver, status(enum), priority?, related_area_id?, related_branch_id?,
  related_agent_id?, submitted_at?, decided_by?, decided_at?, approval_comment?,
  created_at, updated_at`.
- **request_approval** — `id, request_id, actor_id, action, from_status,
  to_status, comment, created_at`.
- **request_attachment** — `id, request_id, kind, storage_path, filename,
  uploaded_by, created_at`.
- **request_activity** — `id, request_id, actor_id, type, from_value?, to_value?, created_at`.
- **business_trip_request_detail** (1:1) — traveler_name/role, purpose,
  justification, from_city, to_city, country, start_date, end_date, num_days,
  travel_type(enum), transportation_type(enum), hotel_required, accommodation,
  est_flight/hotel/transport/per_diem/other, total_estimated, currency('SAR');
  actuals/report later.
- **expense_request_detail** (1:1) — category(enum), expense_date, amount,
  currency('SAR'), description, related_trip_request_id?.
- **leave_request_detail** (1:1) — leave_type(enum), start_date, end_date,
  num_days, reason?, cover_person_id?.
- Enums: `request_type`, `request_status`, `travel_type`, `transportation_type`,
  `expense_category`, `leave_type`.

### 3. Visibility rules (requests)
A user sees a request iff: creator (requested_by) · assigned_approver · is_global
(company_manager sees all) · request's related_area in their scope · or is_admin.

### 4. RLS policies
- `request` SELECT `using (can_see_request(request))` (SECURITY DEFINER helper).
  INSERT `with check (requested_by = auth.uid())`. UPDATE: creator while `draft`;
  approver / company_manager / admin for decisions (status transitions) — server
  action enforces the transition + writes `request_approval`.
- detail / approval / attachment / activity: visible iff parent request visible.

### 5. Approval flow (MVP, configurable later)
On submit, `assigned_approver` set by requester role: Area Manager → Company
Manager; Company Manager → Admin/Director (placeholder); Admin → Company Manager.
Each decision writes `request_approval` and updates `request.status`.

### 6. Calendar & Notifications connection
- Calendar: business-trip (start–end) and leave (start–end) feed the same
  calendar query; status drives color. No extra table needed.
- Notifications: reuse `notification` (from 0006) with `related_request_id`;
  triggers: submitted, approval-required, approved, rejected, cancelled,
  starting-soon, trip-report-required, missing-receipt.
- Task link: on trip approval, optionally auto-create follow-up `task`s
  (book flight/hotel, agenda, trip report, expenses).

### Sidebar / top-bar impact
- Sidebar already shows **Requests** (stub). Later sub-screens: All Requests ·
  Business Trips · Expenses · Leave · Pending Approvals · My Requests.
- Top-bar bell shows combined unread (tasks + requests) notifications.

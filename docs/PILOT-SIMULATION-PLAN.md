# Pilot Simulation Plan — "FreshLine Distribution" (demo FMCG tenant)

**Status:** Plan v1.2 — _design only_. No seeding, no branch creation, no transaction generation, no production. Demo/synthetic data only.
**Purpose:** Validate that every module talks to every other module correctly **and** that real FMCG operational scenarios behave correctly, before approving Phase 1 (Seeder).
**Currency (F1):** the demo tenant operates in **EGP** (the platform's current tenant base currency). All amounts, limits, and accounting figures are EGP; the provider SAR FX seam (`fx-rates.ts`) is platform-tier and out of scope here (see `AUTHORIZATION-MODEL.md` §5.3, §13).
**Cross-cutting:** build order and dependencies are governed by `ARCHITECTURE-ALIGNMENT.md`; approval-driven scenarios sequence after the authorization phases noted inline below.

---

## 0. Objective & chain under test
Stand up one realistic demo tenant with **operational** activity (not empty masters) and prove the full data flow:

```
Sales → Inventory → Warehouse → Finance → Reports → Permissions → Audit
```
…with **zero cross-tenant / cross-branch leakage**, correct accounting, and acceptable performance at FMCG volume.

## 1. Two-track execution model

| Track | Environment | Purpose | Data lifecycle |
|---|---|---|---|
| **A — Automated** | local Postgres via `TEST_DATABASE_URL` (`withRollback` / `actAs` harness) | invariants, accounting integrity, permissions/scope, leakage, audit, idempotency, perf micro-bench | rolled back — nothing persists |
| **B — Manual UAT** | disposable Supabase **branch** DB (+ Vercel preview) | dashboards, reports, approvals, inventory, accounting, workflow, permission visibility, audit readability, FMCG scenarios | persistent, `SIM-*` namespaced, torn down after |

Same seeder + transaction generator feed both — Track A wraps them in a rollback; Track B commits.

## 3. Operational transactions — all via real write paths
Generated by calling the actual server actions / `erp_*` RPCs in business order, exercising numbering, triggers, journals, balances, audit, and workflow:

1. Purchase orders → receipts → **stock ↑, supplier AP ↑, purchase journal**.
2. Rep stock requests → warehouse approval → **warehouse→rep transfer**.
3. Invoices/orders by reps → **stock ↓, customer AR ↑, sales journal, `erp_next_number`**.
4. Over-limit discounts → **workflow approval** → effect on apply. **(P4-dep)**
5. Returns → branch-manager approval → **stock ↑, AR ↓, return journal**. **(P4-dep)**
6. Collections via `erp_record_payment` (+ `idempotency_key`) → **AR ↓, cash journal**; replay to prove idempotency (0118).
7. Supplier settlements → **AP ↓**.
8. Stocktake adjustment + near-expiry write-off. **(near-expiry portion = batch/expiry-dep)**

| Run | Customers | Invoices | Returns | Payments | POs |
|---|---|---|---|---|---|
| **Smoke** | 200 | ~800 | ~80 | ~500 | ~10 |
| **Full** | 1,000 | ~5,000 | ~500 | ~3,000 | ~40 |

Correctness on Smoke is the gate before scaling to Full.

> **Approval-scenario sequencing (F3).** The built approval flows today are **customer onboarding, customer change request, and credit-limit request** (existing `HANDLERS`). **Amount-based** routing for **discount / return / writeoff** (#4, #5, and writeoff in #8) requires **Authorization Model Phase 4** — conditional/amount triggers + new handlers (`AUTHORIZATION-MODEL.md` §9.1). Until Phase 4 lands, the simulation validates approvals on the **built** flows and treats discount/return/writeoff approval routing as **Phase-4-gated** (run those scenarios only once Phase 4 is in the environment). The *non-approval* effects of returns (stock ↑, AR ↓, journal) are validated immediately; only the **approval routing** step is gated.

## 4. Core validation checklist (→ the 8 confirmations)

| # | Confirmation | Check | Track |
|---|---|---|---|
| 1 | Right module updated | per-txn before/after deltas (stock, AR/AP, journal) | A |
| 2 | Stock correct | opening + receipts + returns − sales − adj = `erp_stock`; no illegal negatives | A |
| 3 | Customer balances correct | AR = Σinvoices − Σpayments − Σreturns per customer; control = subledger | A |
| 4 | Sales/returns in dashboards | dashboard/report totals = SQL aggregates | B (+A on query layer) |
| 5 | Scope visibility | `actAs(persona)` visible-row counts = allowed scope only | A + B |
| 6 | Approvals & audit | every gated action → workflow instance + `erp_audit_logs` row; no silent change | A + B |
| 7 | **No leakage** | FreshLine user sees **0** RivalCo rows; rep sees 0 other-branch rows — across customers/invoices/orders/returns/payments/stock; **and each dashboard/widget data source returns only in-scope rows** (widgets are never a data bypass — assert per-widget, not only per-entity) | A (hard gate) |
| 8 | Volume | k6 on lists/search/dashboard at full volume, **including the heaviest-persona dashboard** (RSM: `region` + `own_team` transitive-closure scope) and its widget bundle, which is the worst case for query fan-out × scope closure | A/B |

**Cross-cutting — accounting integrity:** balanced-journal invariant (Σdebit = Σcredit, 0069) after every posting; AR/AP control reconcile to subledgers.

## 6. Success / failure criteria
- **Pass:** 100% invariant + accounting checks green; **zero** leakage/out-of-scope rows; every gated action has matching workflow + audit; dashboards reconcile to ledger; **p95 < 500 ms**, error < 1%, no full-table scans.
- **Fail (any):** balance/stock/journal mismatch · any leak · missing audit/approval · dashboard ≠ ledger · perf budget breach.
- **FMCG scenarios:** see §9 — each has its own pass criteria; capability gaps are *documented*, not counted as correctness failures (see §9.5).

---

## 7. Automated vs manual
- **Automated (Track A):** seeder, transaction generator, invariant/reconciliation asserts, scope/leakage matrix (extends `customer-scope`/`rls`), accounting (extends `accounting.test.ts`), idempotency replay (extends `payment-idempotency.test.ts`), FMCG-KPI asserts (§9.4), k6 (extends `scripts/loadtest/`).
- **Manual (Track B):** dashboards/reports per role, login-as-rep/manager scope visibility, approval UX, audit-log readability, near-expiry alert UX, promotion/claim UX (where present).

## 9. FMCG-Specific Validation

Validates that the system reflects **real FMCG field & trade operations**, not just generic ERP correctness. Each area lists: what to seed, the KPI/scenario definitions, how it's validated, the track, and the **capability dependency** (so the simulation doubles as an FMCG gap analysis — §9.5).

### 9.1 Route Coverage
**Data sources:** `erp_visits` (rep · customer · date · outcome), `erp_invoices`, routes (`route.rep_id`), visit plans.
**Seed:** per rep, a daily **planned route** (the customers on their route); execute visits via the real visit-logging path; a configurable subset are **productive** (generate an order/invoice), the rest are non-productive; leave some planned customers **unvisited**.

| KPI | Definition |
|---|---|
| Coverage % | visited planned customers ÷ total planned customers (per route/day) |
| Productive Calls | visits that produced ≥1 order/invoice |
| Strike Rate | productive calls ÷ total calls |
| Visited vs Non-Visited | counts of each, per route/rep/day |
| Route Completion | completed planned stops ÷ planned stops |

**Validation:** compute each KPI from seeded visit+invoice data and assert it equals the engineered expected value; confirm the **same numbers appear in the rep/supervisor dashboard** (Track B). **Scope:** a rep sees only their own route coverage; a supervisor sees their team's; a branch/regional manager sees their geography — assert via `actAs`.
**Dependency:** `erp_visits` + visit planning (`field.sales`). Confirm exact write path in Phase 0.

### 9.2 Near-Expiry Management
**Precondition:** batch/lot + expiry tracking on stock. Phase 0 confirms whether lot-level expiry is modeled; if not, this section becomes a **documented gap** (§9.5) rather than a failed test.
**Shelf-life profiles:** tag products/batches with **4-day, 15-day, 45-day, 90-day** shelf lives; seed batches with manufacture/expiry dates straddling each profile's alert threshold (just inside, just outside, and already-expired).

| Scenario | Validation |
|---|---|
| Near-expiry alert | alert fires when days-to-expiry ≤ profile threshold; does **not** fire outside it |
| Expiry returns | returns flagged "expiry" increase stock then route to write-off |
| Write-off | near-expiry/expired stock written off → **stock ↓ + expense/loss journal** posted |
| Stock impact | post-scenario `erp_stock` and journal deltas match expected |
| FEFO (if supported) | picking consumes earliest-expiry batch first |

**Track:** A (thresholds, stock/journal deltas) + B (alert UI, near-expiry report).

### 9.3 Trade Spend & Promotions
**Precondition:** promotion/free-goods engine + trade-spend/claims. **Designed-only today** (no production code): the canonical design is `FMCG-HIERARCHY-CUSTOMER-PROMO-PLAN.md` (planned `erp_promotions`; types `percent_discount`, `quantity_free`, `bundle`, `customer_specific`; `trade_spend` / `expected_roi` / `actual_roi`), with a reserved promotion hook in the pricing engine (`0106`, "S5 hook"). The legacy `0002_promotions.sql` table is **not tenant-scoped and unusable**. Phase 0 confirms current state; where present → validate; where absent → documented FMCG gap feeding the roadmap (promotions engine → Approval Matrix → Industry Packs), **not** a correctness failure.

**Naming alignment (F6):** this simulation's "10+1 / 2+1" map to the promo plan's `quantity_free` type (the plan's examples are 5+1 / 10+2 — same mechanic, different ratios). **Claims** are an *addition* of this simulation not yet in the promo plan or code; treat trade-spend **claim generation** as a documented future item (the promo plan currently models `trade_spend` + ROI but no claims workflow). Validation here must conform to `erp_promotions` once built — no parallel promotion shape.

| Scenario | Validation (where capability exists) |
|---|---|
| Free goods **10+1** | qualifying qty triggers 1 bonus unit; bonus billed at zero; **stock ↓ incl. free units**; COGS-on-free-goods journal correct |
| Free goods **2+1** | same mechanics at the 2+1 ratio |
| Display / listing support | trade-spend agreement accrues a liability/expense |
| Approval workflow | promotion/claim routed for approval (per authorization spec limits) |
| Claim generation | per-customer/period claim computed from qualifying volume |
| ROI / effectiveness | incremental sales vs spend computed and surfaced |

**Track:** A (free-goods qty/stock/journal math) + B (promotion setup, approval, claim, ROI views).

### 9.4 FMCG Sales KPIs
**Data sources:** `erp_invoices` (+ lines), `erp_customers`, `erp_products_catalog`, routes, `erp_rep_targets`.

| KPI | Definition | Track |
|---|---|---|
| Active customers | distinct customers with ≥1 invoice in the period | A + B |
| New customers | customers created within the period | A + B |
| Sales per rep | Σ net sales grouped by `salesman_id` | A + B |
| Sales per route | Σ net sales grouped by `route_id` | A + B |
| **Target achievement (F5)** | actual net sales ÷ `erp_rep_targets.target_amount` for the period, per rep — **buildable now** (the table exists; only the calc/widget is unbuilt, not the data) | A + B |
| SKU distribution | distinct SKUs sold ÷ active SKUs (overall and per outlet) | A |
| Numeric distribution | % of outlets that purchased a given SKU (outlets selling SKU ÷ total outlets) | A |
| Weighted distribution | **future-ready** — sales-weighted distribution; requires a market-universe baseline; documented as future, formula reserved | doc |

**Validation:** compute each KPI from seeded transactions, assert against engineered expectations, and confirm the **dashboard/reports show identical figures**; all KPIs respect scope (rep sees own, manager sees geography/team).

### 9.5 Capability dependency & gap-analysis rule
The FMCG validation doubles as a **gap analysis**:
- Capabilities that **exist** are validated to the criteria above and must pass.
- Capabilities **not yet built** (e.g. promotion engine, lot-expiry, weighted distribution) are recorded in a **gap register** with the scenario that would validate them — feeding the roadmap (Workflow Engine → Approval Matrix → Industry Packs).
- A missing capability is a **documented gap, not a correctness failure** — this respects the feature freeze and keeps the correctness gate (§4) clean.
Phase 0 produces the dependency confirmation: which of §9.1–9.4 are testable now vs. recorded as gaps.
